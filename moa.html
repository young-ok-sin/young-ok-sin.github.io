MoA Case Study

상태 전이를 기준으로 금융 계획 흐름을 안정화한 백엔드 설계

MoA는 소비 분석과 목표 기반 예산·저축 계획을 제공하는 개인 금융 서비스입니다. 저는 Plan 도메인 백엔드를 맡아 목표 생성 이후의 상태 전이, 우선순위 정렬, 비동기 콘텐츠 생성과 알림 연동까지 이어지는 핵심 사용자 흐름을 구현했습니다.

Service Goal

사회 초년생이 자신의 소비 습관을 분석하고, 목표 중심으로 예산과 저축 계획을 세울 수 있도록 돕는 서비스입니다. 계획을 세우는 순간보다, 이후 상태가 안정적으로 이어지는 경험이 중요하다고 판단했습니다.

My Ownership

  • Plan 상태 전이 및 우선순위 재정렬 로직 구현
  • AI 이미지·영상 생성 비동기 파이프라인 설계
  • Polling scheduler 기반 생성 상태 추적
  • AWS S3 공통 저장 추상화와 FCM 알림 연동

Why It Mattered

사용자는 목표를 생성한 뒤 즉시 결과가 완성되지 않아도, 현재 어디까지 진행됐는지 신뢰할 수 있어야 합니다. 그래서 이 프로젝트는 기능 구현보다 상태를 끝까지 추적하는 설계가 더 중요했습니다.

Problem Solving

생성 흐름을 실패로 오인하지 않도록 상태 전이를 분리했습니다

01

Problem

영상 생성 서버가 queued 또는 processing 상태를 반환하면, 이후 상태를 안정적으로 추적하지 못해 처리 중 작업도 실패처럼 보이는 문제가 있었습니다.

02

Cause

생성 요청과 완료 처리가 하나의 흐름에 강하게 묶여 있었고, 작업 상태를 DB에 충분히 남기지 않아 즉시 완료되지 않는 요청을 이어서 추적할 기준점이 부족했습니다.

03

Solution

생성 메타데이터를 Plan 엔티티에 저장하고, 요청과 완료 처리를 분리했습니다. queued와 processing은 실패가 아니라 진행 중 상태로 유지한 뒤, polling scheduler가 주기적으로 조회해 ready, error, cancelled로 전이하도록 설계했습니다.

동시에 idempotency key와 상태 전이 메서드를 추가해 중복 요청과 경합 상황에서도 같은 작업을 일관되게 추적할 수 있게 했습니다.

04

Result

즉시 완료되지 않는 생성 작업을 실패로 오인하지 않게 되었고, 다중 목표 생성 상황에서도 처리 중 상태를 안정적으로 노출할 수 있게 됐습니다. 이후 스케줄러와 목록 조회 테스트를 추가하며 운영 리스크도 함께 줄였습니다.

Tech Decisions

기술 결정은 유지보수성과 운영 안정성 기준으로 정리했습니다

Storage Abstraction

공통 S3 저장 계층

member와 plan에서 반복되던 업로드, URL 생성, 삭제 흐름을 공통 계층으로 추상화해 저장소 관련 코드를 단순하게 유지했습니다.

Async Pipeline

생성 요청과 완료 처리 분리

목표 생성 이후 AI 이미지와 영상을 순차적으로 연결하되, 결과 저장과 상태 갱신을 분리해 실패 지점을 좁히고 추적 가능성을 높였습니다.

Operational Stability

Polling 기반 상태 추적

외부 생성 서버의 응답 지연을 흡수하기 위해 polling scheduler와 상태 전이 로직을 분리해 운영 안정성을 높였습니다.

User Feedback

FCM 완료 알림

비동기 생성이 끝나는 시점에 결과를 바로 확인할 수 있도록 FCM 알림을 연동해 사용자 경험을 보완했습니다.

Outcome

What I Learned

백엔드 포트폴리오에서 중요한 것은 API 개수보다, 예외 상황과 지연 상태를 어떻게 설계하고 설명할 수 있는지라고 생각합니다. MoA는 그 기준을 가장 명확하게 보여주는 프로젝트였습니다.
