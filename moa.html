Problem Solving

Problem / Cause / Solution / Result

01 Problem

영상 생성 서버가 queued 또는 processing 상태를 반환하면, 이후 상태를 안정적으로 추적하지 못해 처리 중 작업도 실패처럼 보이는 문제가 있었습니다.

02 Cause

생성 요청과 완료 처리가 하나의 흐름에 강하게 묶여 있었고, 작업 상태를 DB에 충분히 남기지 않아 즉시 완료되지 않는 요청을 이어서 추적할 기준점이 부족했습니다.

03 Solution

생성 메타데이터를 Plan 엔티티에 저장하고, 요청과 완료 처리를 분리했습니다. queued와 processing은 실패가 아니라 진행 중 상태로 유지한 뒤, polling scheduler가 주기적으로 조회해 ready, error, cancelled로 전이하도록 설계했습니다.

04 Result

즉시 완료되지 않는 생성 작업을 실패로 오인하지 않게 되었고, 다중 목표 생성 상황에서도 처리 중 상태를 안정적으로 노출할 수 있게 됐습니다. 이후 스케줄러와 목록 조회 테스트를 추가하며 운영 리스크도 함께 줄였습니다.

Tech Decisions

Implementation notes

Storage Abstraction

member와 plan에서 반복되던 업로드, URL 생성, 삭제 흐름을 공통 계층으로 추상화해 저장소 관련 코드를 단순하게 유지했습니다.

Async Pipeline

생성 요청과 완료 처리를 분리해 실패 지점을 좁히고, 결과 저장과 상태 갱신을 독립적으로 다루도록 구성했습니다.

Polling Scheduler

외부 생성 서버의 응답 지연을 흡수하기 위해 polling scheduler와 상태 전이 로직을 분리해 운영 안정성을 높였습니다.

FCM Feedback

비동기 생성이 끝나는 시점에 사용자에게 바로 결과를 전달할 수 있도록 FCM 완료 알림을 연동했습니다.
