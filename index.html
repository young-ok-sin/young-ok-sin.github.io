Problem Solving

03

생성 상태를 끝까지 추적한 파이프라인 설계

01

Problem

영상 생성 서버가 queued 또는 processing 상태를 반환하면, 백엔드가 이후 상태를 안정적으로 추적하지 못했습니다. 여러 목표가 동시에 생성될 때 아직 진행 중인 작업도 실패처럼 보이는 문제가 있었습니다.

02

Cause

생성 요청과 완료 처리가 하나의 흐름으로 강하게 묶여 있었고, 작업 상태를 DB에 충분히 남기지 않았습니다. 그래서 즉시 완료되지 않는 작업은 후속 추적의 기준점이 부족했습니다.

03

Solution

생성 메타데이터를 Plan 엔티티에 저장하고 요청과 완료 처리를 분리했습니다. queued와 processing 상태는 실패로 처리하지 않고 DB에 유지한 뒤, polling scheduler가 주기적으로 상태를 조회해 ready, error, cancelled로 전이하도록 설계했습니다.

동시에 idempotency key와 상태 전이 메서드를 추가해 중복 요청과 경합 상황에서도 같은 작업을 일관되게 추적할 수 있도록 했습니다.

04

Result

즉시 완료되지 않는 생성 작업을 더 이상 실패로 오인하지 않게 되었고, 다중 목표 생성 상황에서도 처리 중 상태를 안정적으로 노출할 수 있게 되었습니다. 이후 scheduler와 목록 조회 테스트를 추가하며 운영 리스크도 함께 줄였습니다.
