Project

Rawfishs

Rawfishs는 실시간 회의 진행 데이터를 관리하고, 회의 종료 후 AI 기반 회의 요약과 action item을 생성·조회할 수 있게 만든 협업 서비스 백엔드 프로젝트입니다.

Overview

Rawfishs는 사용자가 회의를 생성·참여하고, 안건과 타이머를 실시간으로 관리할 수 있도록 만든 협업 서비스입니다. 백엔드에서는 회의 중 상태와 참가자 정보를 Redis에 유지하고, WebSocket 이벤트로 타이머와 진행 상태를 동기화했습니다. 회의가 종료되면 Redis에 저장된 안건과 STT 데이터를 정리해 DB와 S3에 반영하고, 이어서 AI 서버에 회의록 생성을 요청하도록 구성했습니다.

핵심은 회의 중 실시간 상태 처리와 회의 종료 후 비동기 AI 후처리를 한 흐름으로 연결하면서도, 실패 시 종료 자체는 깨지지 않게 분리하는 것이었습니다.

이후 히스토리 API에서는 회의 요약, 핵심 의사결정, 안건별 action item, STT 스크립트까지 조회할 수 있도록 구성해 실시간 협업과 사후 기록 관리가 하나의 서비스 경험으로 이어지게 했습니다.

Tech Stack

Area Stack Reason
Backend API / Realtime Spring Boot, WebSocket, LiveKit 회의 생성·참여 API, 회의 입장 토큰 발급, 실시간 타이머와 상태 이벤트를 안정적으로 연결하기 위해 사용
Storage / State Spring Data JPA, MySQL, Redis, AWS S3 회의 히스토리와 리포트를 영속 저장하고, 진행 중 상태는 Redis에 유지하며, STT 스크립트는 파일 형태로 S3에 저장하기 위해 사용
External Integration RestClient, AI server, Resilience4j 회의 종료 후 AI 서버에 리포트 생성을 요청하고 timeout, retry, circuit breaker로 외부 장애가 전체 흐름을 깨지 않게 제어하기 위해 사용

Key Features

  • 회의 생성·참여 시 Redis에 회의와 참가자 상태를 저장하고 입장 토큰을 발급했습니다.
  • WebSocket 기반 타이머 start, stop, sync 기능을 구현하고 호스트 권한 검증을 함께 처리했습니다.
  • 회의 종료 시 Redis 안건 snapshot을 DB에 저장하고 STT 스크립트를 S3에 업로드하도록 구성했습니다.
  • AI 서버 연동으로 회의 요약, 핵심 결정, 안건별 summary와 action item을 생성하고 저장했습니다.
  • 회의 히스토리 API에서 회의 시간, 참가자 수, action item, STT 스크립트까지 함께 조회할 수 있게 했습니다.

개발 내용

Redis와 WebSocket 기반 실시간 회의 상태 관리

회의 생성·참여, 참가자 상태, 타이머 이벤트를 실시간으로 다루기 위해 Redis와 WebSocket 중심으로 상태 흐름을 정리했습니다.

  • 회의와 참가자 상태를 Redis에 저장하고, 입장 토큰 발급과 실시간 이벤트 흐름을 연결했습니다.
  • 타이머 start, stop, sync 이벤트와 호스트 권한 검증을 함께 처리했습니다.
  • 서버 메모리와 Redis를 함께 활용해 타이머 종료 직후에도 상태 보정 메시지를 보낼 수 있게 했습니다.

회의 종료 후 AI 리포트 생성 파이프라인 구성

회의 종료 시점의 Redis 데이터를 그대로 버리지 않고, DB와 S3 저장 이후 AI 리포트 생성으로 이어지는 후처리 흐름을 만들었습니다.

  • Redis 안건 snapshot 저장, AI 리포트 요청, 정리 작업 순서로 종료 비동기 흐름을 재구성했습니다.
  • AI 호출 실패가 회의 종료 자체를 막지 않도록 후처리 로직을 분리했습니다.
  • Resilience4j 기반으로 timeout, retry, circuit breaker를 적용해 외부 장애 대응을 보강했습니다.

히스토리 조회와 안건 단위 리포트 매핑 구현

회의 요약만 보여주는 데 그치지 않고, 안건별 summary와 action item, STT 스크립트까지 조회할 수 있도록 히스토리 API를 확장했습니다.

  • Agenda 엔티티에 `redisAgendaId`를 저장해 AI 응답의 agendaKey와 DB 안건을 정확히 매핑했습니다.
  • 회의 시간, 참가자 수, action item, STT 스크립트를 함께 조회할 수 있는 히스토리 응답 구조를 만들었습니다.
  • 실시간 회의 상태와 종료 후 기록 조회가 하나의 서비스 흐름으로 이어지도록 API를 정리했습니다.

Retrospective

  • Redis 같은 실시간 저장소와 DB를 함께 쓰는 구조에서는 식별자 일관성을 먼저 설계해야 후처리 오류를 줄일 수 있었습니다.
  • 외부 AI 호출은 성공 경로보다 실패와 재시도 정책을 먼저 설계해야 전체 서비스 안정성이 높아진다는 점을 경험했습니다.
  • 회의 종료처럼 후처리가 많은 흐름은 비동기 처리와 정리 순서를 명확히 나누는 것이 유지보수와 장애 대응에 유리했습니다.
