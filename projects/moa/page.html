Overview

MoA는 사회 초년생이 자신의 소비 흐름을 분석하고 목표 기반 예산을 세울 수 있도록 돕는 서비스입니다. 저는 Plan 도메인에서 목표 생성 이후의 상태 전이와 생성 작업 추적 흐름을 설계했습니다.

핵심은 “생성 요청을 성공시켰는가”가 아니라 “생성 중 상태를 안정적으로 보여줄 수 있는가”였습니다.

Tech Stack

Area Stack Reason
Backend Java, Spring Boot Plan 상태 관리와 API 흐름을 안정적으로 구성
Async polling scheduler, fcm 생성 중 상태 추적과 완료 피드백 전달
Storage AWS S3 공통 파일 저장과 URL 관리 추상화

Key Features

  • Plan 상태 전이와 우선순위 재정렬 로직 구현
  • state-transition 기준으로 생성 요청과 완료 처리를 분리
  • polling 기반으로 queued, processing, ready 상태를 추적
  • 완료 시 fcm 알림과 S3 저장 흐름을 연결

Troubleshooting

Problem

외부 생성 서버가 즉시 완료되지 않으면 사용자에게 실패처럼 보이는 문제가 있었습니다.

Solution

요청과 완료 처리를 분리하고, polling scheduler가 진행 상태를 주기적으로 갱신하도록 바꿨습니다. 또한 생성 메타데이터를 엔티티에 저장해 재조회 기준을 명확히 했습니다.

Result

생성 중 상태를 안정적으로 노출할 수 있게 되었고, 다중 목표 생성 상황에서도 UX가 일관되게 유지됐습니다.

Retrospective

  • 상태 전이 설계는 비즈니스 규칙보다 UX 안정성과 함께 봐야 한다.
  • 비동기 작업은 완료보다 추적 방식이 먼저 설계되어야 한다.
  • 공통 저장 흐름은 초기에 추상화할수록 후속 작업이 단순해진다.
