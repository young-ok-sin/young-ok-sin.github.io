Project

MoA

금융 데이터 기반 목표 관리 서비스에서 목표 생성 이후 AI 후처리까지 이어지는 흐름을 설계한 백엔드 프로젝트입니다.

Overview

MoA는 사용자의 금융 데이터를 바탕으로 목표를 만들고, 예산 배분과 저축 진행 상황을 관리하는 서비스입니다. 저는 목표 도메인 백엔드를 맡아 목표 생성 이후의 상태 전이, AI 이미지·영상 생성 후처리, 그리고 이 과정에서 발생하는 정합성 문제를 다뤘습니다.

핵심은 목표를 저장하는 것보다, 목표 생성 이후 이어지는 비동기 작업을 끊기지 않게 연결하고 상태를 안정적으로 보여주는 것이었습니다.

특히 외부 AI 서버와 연결되는 흐름에서는 요청-응답, 장시간 작업 추적, 상태 업데이트가 서로 엉키지 않도록 설계해야 했습니다. 그래서 목표 생성 API 자체보다 생성 이후의 후처리 흐름과 상태 일관성에 더 많은 비중을 두고 구현했습니다.

Tech Stack

Area Stack Reason
Backend Java, Spring Boot 3, JPA, Hibernate 목표 생성, 상태 전이, 예산 반영 로직을 서비스 계층 중심으로 구성
Async Spring Async, TransactionalEventListener, Scheduler, polling 생성 요청과 AI 후처리를 분리하고 장시간 작업 상태를 추적
Integration FCM, S3, external AI image server, external video server 생성 완료 알림, 결과 저장, 외부 서버 연동을 하나의 흐름으로 연결

Key Features

  • 목표 생성 API를 구현하면서 단순 저장에 그치지 않고 우선순위 이동, 예산 재배치, 즉시 저축 반영까지 하나의 흐름으로 묶었습니다.
  • 목표 생성 이후 AI 이미지·영상 생성은 동기 처리하지 않고 이벤트 기반 비동기 파이프라인으로 분리했습니다.
  • 영상 생성은 polling scheduler로 추적했고, PROCESSING, SUCCEEDED, FAILED 상태를 나눠 장시간 작업을 관리했습니다.
  • 목표 완료/지연 상태 반영과 FCM 알림 전송까지 연결해 생성 이후 사용자 경험이 끊기지 않도록 구성했습니다.

개발 내용

목표 생성 이후 비동기 후처리 파이프라인 분리

목표 생성 API가 외부 AI 작업까지 직접 기다리지 않도록 트랜잭션 이후 비동기 후처리 흐름으로 분리했습니다.

  • `TransactionalEventListener`와 `@Async`를 사용해 커밋 이후 후처리 작업이 이어지도록 구성했습니다.
  • 외부 영상 서버 연동에는 retry와 idempotency key를 적용해 중복 생성 없이 재시도할 수 있게 했습니다.
  • 생성 API는 핵심 저장 책임에 집중하고, 후처리는 별도 흐름으로 이어지도록 경계를 나눴습니다.

장시간 AI 작업 상태를 polling scheduler로 추적

영상 생성처럼 완료 시점을 예측하기 어려운 작업은 별도 scheduler로 추적해 상태를 사용자에게 안정적으로 보여주도록 했습니다.

  • PROCESSING, SUCCEEDED, FAILED 상태를 분리해 장시간 작업을 관리했습니다.
  • 완료 시점에 저장과 알림 흐름이 자연스럽게 이어지도록 후속 작업을 연결했습니다.
  • 목표 생성 이후에도 사용자가 현재 상태를 확인할 수 있도록 조회 흐름을 정리했습니다.

상태 정합성과 동시성 제어 보강

목표 생성과 AI 상태 업데이트가 겹치는 상황에서도 오래된 데이터가 최신 상태를 덮어쓰지 않도록 동시성 제어를 넣었습니다.

  • `@Version` 기반 낙관적 락과 조건부 update를 적용해 stale update 가능성을 줄였습니다.
  • 목표 기간 연장과 완료 목표 이력 보존에 필요한 보강 로직을 함께 정리했습니다.
  • 비동기 상태 변경 이후에도 조회 응답이 일관되도록 상태 전이 흐름을 다듬었습니다.

Retrospective

  • 외부 연동이 포함된 기능은 요청-응답 흐름과 후처리 흐름을 분리해야 안정적으로 운영할 수 있다는 점을 확인했습니다.
  • 상태가 많은 도메인은 기능 추가보다 상태 전이와 정합성 설계가 먼저라는 점을 배웠습니다.
  • 동시성 문제는 나중에 보완하기보다 version과 상태 조건을 초기에 함께 설계하는 편이 안전했습니다.
